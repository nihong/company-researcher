# 回测与跟踪验证模块 (Tracking & Backtesting)

任何量化策略如果没有经过实战验证，就只是"看起来很美的假说"。本模块为 `company-researcher` 建立一套**事后追踪机制**，用真实的市场结果来检验每一份报告的预判能力，并据此迭代打分模型的权重。

---

## 一、每份报告必须登记的"快照" (Snapshot)

在每份投研报告（或量化打分战报）交付时，**必须同时生成一条跟踪记录**，追加写入 `tracking/ledger.csv` 文件。

### CSV 字段定义

```csv
日期,股票代码,股票名称,报告时股价,报告时PE,量化评分,评级,买方决策,30日后股价,60日后股价,90日后涨跌幅,决策是否正确,复盘备注
```

### 字段说明

| 字段 | 来源 | 说明 |
|------|------|------|
| 日期 | 报告日期 | YYYY-MM-DD |
| 股票代码 | 报告标的 | 如 688239 |
| 股票名称 | 报告标的 | 如 航宇科技 |
| 报告时股价 | `opencli eastmoney quote` | 报告生成当天的收盘价 |
| 报告时PE | `opencli eastmoney quote` | 动态市盈率 |
| 量化评分 | A_Share_Quant_Scorer 输出 | 0-100 分 |
| 评级 | 量化战报 | S/A/B/C/D |
| 买方决策 | 报告结论 | 左侧布局/右侧跟随/观望等待/坚决规避 |
| 30日后股价 | **待回填** | 报告发布30个自然日后，通过 opencli 获取并回填 |
| 60日后股价 | **待回填** | 报告发布60个自然日后回填 |
| 90日后涨跌幅 | **待回填** | (90日后股价 - 报告时股价) / 报告时股价 × 100% |
| 决策是否正确 | **待回填** | 根据涨跌幅判断：S/A级且上涨=正确，B/C级且下跌=正确 |
| 复盘备注 | **人工/AI填写** | 简述为什么对/错，哪个因子失效了 |

---

## 二、回填流程 (Backfill SOP)

### 触发方式
用户可随时说："**回测一下之前的报告**"或"**复盘上个月的决策**"，触发回填流程。

### 执行步骤

1. **读取 `tracking/ledger.csv`**，筛选出"30日后股价"为空且距今已超过30天的记录。
2. **通过 `opencli eastmoney kline` 获取对应日期的历史收盘价**，填入30/60/90日字段。
   ```bash
   opencli eastmoney kline <股票代码> --period day --count 90 -f json
   ```
3. **自动判断决策正确性**：
   - S/A 级（买入信号）：90日后涨幅 > 0% → ✅ 正确
   - S/A 级（买入信号）：90日后跌幅 > 10% → ❌ 严重失误
   - B 级（观望信号）：90日后涨幅 > 20% → ❌ 踏空失误（安全边际设置过严）
   - C/D 级（规避信号）：90日后跌幅 > 0% → ✅ 正确
4. **生成复盘报告**（Markdown 表格），统计：
   - 整体胜率（正确决策数 / 总决策数）
   - 各评级的平均收益率
   - 最大的失误案例分析

## 三、权重迭代规则 (Weight Calibration)

当累计样本量达到 **20 条以上**时，必须启动权重迭代审计：

### 审计方法

1. **计算各维度的"因子有效性"（IC 值近似）**：
   - 对于每个维度（基本面、估值、资金等），将其维度得分与90日后涨跌幅做相关性排序。
   - 相关性越强的维度，说明预测能力越好，权重应当提升。
   - 相关性为负或接近零的维度，说明该因子失效，权重应当降低。

2. **输出权重调整建议**：
   ```
   [权重迭代建议] 基于过去 N 条样本：
   - 资金与筹码维度：相关性最高 (r=0.72)，建议权重从 20% → 25%
   - 估值安全垫维度：相关性最低 (r=0.08)，建议权重从 25% → 15%
   - 政策红利维度：相关性中等 (r=0.45)，维持当前权重
   ```

3. **人工确认后更新** `agents/A_Share_Quant_Scorer.md` 中的权重配置。
